## Directory Conventions

| Path | Purpose |
|------|---------|
| `configs/<name>/` | Shareable configurations (Biome, TypeScript, Renovate, ESLint, etc.) |
| `packages/<name>/` | Runtime libraries (React components, hooks, utilities, etc.) |

Both are automatically discovered by pnpm and NX — no manual workspace registration needed. `pnpm-workspace.yaml` already includes `configs/*` and `packages/*`.

Within a package, follow the existing convention:

```
configs/my-config/
├── LICENSE         ← MIT (copy from any sibling package)
├── README.md       ← Usage instructions
├── package.json    ← See below
├── default.json    ← The entry point (config files) — or src/index.ts for code packages
└── CHANGELOG.md    ← Auto-generated by NX after the first release
```

## Step 1 — Create the Package Directory

```bash
mkdir -p configs/my-config   # or packages/my-lib
```

## Step 2 — Write package.json

Use an existing sibling package as a template. Here are the critical fields:

```json
{
  "name": "@tutods/my-config",
  "version": "0.0.1",
  "private": false,          // ← Must be false for publishing
  "license": "MIT",
  "publishConfig": {
    "access": "public"
  },
  "repository": {
    "type": "git",
    "url": "ssh://git@github.com:tutods/lib.git",
    "directory": "configs/my-config"   // ← Must match the actual path
  },
  "files": [
    "default.json",
    "README.md",
    "LICENSE"
  ]
}
```

| Field | Why it matters |
|-------|---------------|
| `private: false` | Without this, npm refuses to publish |
| `publishConfig.access: "public"` | Required for scoped packages (`@tutods/*`) |
| `repository.directory` | Must match the subdirectory path exactly — npm provenance depends on it |
| `files` | Controls what lands in the tarball — keep it minimal |
| `version` | Start at `0.0.1` — NX will bump it from the first conventional commit |

## Step 3 — Add Package Content

Write your config files, source code, or whatever the package exports. For a config package, the entry point is usually a JSON file. For code packages, add a `rslib` build target (copy from `packages/components`).

## Step 4 — Create the Baseline Git Tag

**This is the step that trips people up.** NX resolves the "current version" of each package from git tags matching the pattern `{projectName}@{version}`. Without a tag, NX silently skips the package during `nx release`.

After pushing your initial commit to `main`, create the baseline tag:

```bash
# Find the commit hash where version 0.0.1 was first on main
COMMIT=$(git log --oneline main | head -1 | cut -d' ' -f1)

# Create and push the tag
git tag @tutods/my-config@0.0.1 "$COMMIT"
git push origin @tutods/my-config@0.0.1
```

> The tag name must be `@{scope}/{name}@{version}` — exactly matching the `releaseTagPattern` in `nx.json`: `{projectName}@{version}`.

## Step 5 — Commit and Trigger the Release

1. Commit your package with a conventional commit that triggers a bump:

   ```bash
   git add configs/my-config/
   git commit -m "feat(my-config): initial release of @tutods/my-config"
   git push
   ```

2. If the `release-pr` workflow doesn't fire automatically (e.g., the path filter missed it), trigger it manually:

   ```bash
   gh workflow run release-pr.yaml
   ```

3. The workflow runs `nx release --skip-publish`, which:
   - Sees `feat(my-config): ...` → resolves `minor` bump
   - Anchors from the tag to determine the current version (`0.0.1`)
   - Bumps to `0.1.0`, generates a `CHANGELOG.md`
   - Pushes to `release/next` and opens a PR

4. Merge the release PR → `release.yaml` publishes to npm via OIDC trusted publishing.

## How Autopublishing Works

```
git push (feat commit on main)
         │
         ▼
release-pr.yaml ──► nx release --skip-publish
         │                   │
         │              bumps versions + changelogs
         │                   │
         │              pushes release/next
         │              creates PR
         │
  [You merge the PR]
         │
         ▼
release.yaml ──► npm publish (OIDC) + git tags + GitHub Releases
```

- **Version bumps** are determined entirely by conventional commits since the last tag.
- **Changelogs** are auto-generated from commit messages.
- **Publishing** uses npm Trusted Publishing (OIDC) — no stored tokens.

## Common Pitfalls

| Problem | Cause | Fix |
|---------|-------|-----|
| NX skips the package during `nx release` | No git tag exists for the package | Create the baseline tag (Step 4) |
| `npm publish` fails with 404 | npm Trusted Publishing not configured for the new package scope | Add the GitHub repo to [npm's Trusted Publishers](https://www.npmjs.com/settings/tutods/publishers) |
| `pnpm install` fails in CI with `ERR_PNPM_IGNORED_BUILDS` | New transitive dependencies need build script approval | Run `pnpm approve-builds` locally or set `allowBuilds` in `pnpm-workspace.yaml` |
| `private: false` but package still won't publish | Missing `publishConfig.access: "public"` for scoped packages | Add the field (Step 2) |
| Version goes `0.0.1` → `0.0.1` (no bump) | Commit type is `chore`, `docs`, `style`, or `test` — these don't bump version | Use `feat` for a minor bump or `fix` for a patch |
